# Minecraft 服务器代理监控系统 —— 极简 MVP 需求规格说明书

## 1. 引言

### 1.1 项目背景
为了快速验证 Minecraft 代理监控的核心概念，我们决定实现一个**最小可行产品（MVP）**。该版本仅针对 Minecraft **1.21.11** 版本（协议版本按当前参考为 **774**），实现最基本的端口隐藏、玩家信息提取、在线展示和封禁功能。目标是 **最短时间内** 产出一个可运行的原型，用于演示和后续迭代。

### 1.2 项目目标
- 构建一个基于 C++ + Workflow 的 Minecraft 代理，仅支持 1.21.11 客户端。
- 隐藏后端服务器端口，将所有流量通过代理转发。
- 解析握手包，获取玩家名和 IP，支持基于用户名和 IP 的封禁。
- 提供简单的 Web 页面，实时显示在线玩家，并可执行封禁/解封操作。

协议版本核验参考：`docs/mc-protocol-reference.md`。

### 1.3 目标用户
- 开发人员（用于测试和演示）
- 熟悉 Minecraft 1.21.11 版本的小型服务器管理员

## 2. 总体描述

### 2.1 产品范围
**包含功能**：
- 代理监听公网端口（如 25565），转发到后端真实服务器（配置文件中指定）。
- 仅支持 Minecraft 1.21.11 客户端连接（协议版本 774）。
- 在握手阶段提取玩家名和源 IP，不解析 UUID 或其他高级字段。
- 内存黑名单（支持用户名和 IP），通过 HTTP API 动态增删。
- 双向数据透传（包括 Ping 请求和游戏内流量），对游戏内容无修改。
- 极简管理页面：展示在线玩家列表、黑名单列表，并提供添加/删除封禁的表单。
- 所有数据仅保存在内存中，不持久化。

**不包含功能**（与标准 MVP 相比的简化）：
- 不支持多版本客户端。
- 无数据库持久化。
- 无统计图表、无 WebSocket、无登录认证。
- 无连接限流、负载均衡等高级特性。
- 前端不处理玩家头像、地理位置等。

## 3. 功能需求

### 3.1 代理核心
| 需求 ID | 功能名称 | 详细描述 | 优先级 |
|--------|---------|---------|--------|
| F1.1   | 端口隐藏 | 代理监听端口 25565（可配置），将流量转发至配置的后端服务器（如 `127.0.0.1:25566`）。 | P0 |
| F1.2   | 协议解析（仅 1.21.11） | 解析客户端发送的 Handshake 包（Packet ID 0x00），提取协议版本、服务器地址、端口、Next State；若 Next State 为登录（2），继续读取 Login Start 包（Packet ID 0x00），获取玩家名。若协议版本不是 774，则直接拒绝连接（可选）。 | P0 |
| F1.3   | 访问控制 | 根据玩家名和源 IP 查询内存黑名单，若命中则立即断开连接（可返回一个拒绝登录的包，但为简化直接断开）。 | P0 |
| F1.4   | 双向转发 | 若允许连接，代理建立到后端的 TCP 连接，并将客户端已接收的数据（可能包含握手包）发送给后端；之后在两个方向上进行数据透传，直到任意一方断开。 | P0 |

### 3.2 监控与管理
| 需求 ID | 功能名称 | 详细描述 | 优先级 |
|--------|---------|---------|--------|
| F2.1   | 在线玩家列表 | 记录当前在线玩家：玩家名、IP、连接时间。存储在内存的线程安全容器中。 | P0 |
| F2.2   | 黑名单管理 | 内存中维护两个集合：封禁的用户名集合、封禁的 IP 集合。支持增删操作，增删时需检查是否影响当前在线玩家（封禁时踢出匹配的在线玩家）。 | P0 |

### 3.3 HTTP 管理 API
| 需求 ID | 功能名称 | 详细描述 | 优先级 |
|--------|---------|---------|--------|
| F3.1   | 获取在线玩家 | `GET /api/players` 返回 JSON 数组，每个元素包含 `name`、`ip`、`since`（Unix 时间戳）。 | P0 |
| F3.2   | 获取黑名单 | `GET /api/bans` 返回两个数组：`names` 和 `ips`。 | P0 |
| F3.3   | 添加封禁 | `POST /api/ban`，请求体 JSON `{"type":"name","value":"Steve"}` 或 `{"type":"ip","value":"1.2.3.4"}`。成功后更新内存黑名单，并立即踢出所有匹配的在线玩家（关闭对应连接）。 | P0 |
| F3.4   | 解除封禁 | `DELETE /api/ban?type=name&value=Steve` 或 `?type=ip&value=1.2.3.4`，从黑名单中移除。 | P0 |

### 3.4 Web 管理页面
| 需求 ID | 功能名称 | 详细描述 | 优先级 |
|--------|---------|---------|--------|
| F4.1   | 页面布局 | 一个简单的 HTML 页面，包含两个主要区域：在线玩家列表（表格）、黑名单列表（两个列表）。 | P0 |
| F4.2   | 在线玩家表格 | 每行显示玩家名、IP、连接时间（格式化为可读时间），每行有一个“封禁”按钮，点击后将该玩家名加入黑名单（并自动踢出）。 | P0 |
| F4.3   | 黑名单展示 | 分别展示封禁的用户名列表和 IP 列表，每个条目旁边有“解封”按钮。 | P0 |
| F4.4   | 添加封禁表单 | 提供两个表单：一个用于输入用户名封禁，一个用于输入 IP 封禁。 | P0 |
| F4.5   | 自动刷新 | 页面每隔 3 秒通过 AJAX 调用 API 刷新在线玩家和黑名单数据。 | P1 |

## 4. 非功能需求

### 4.1 性能需求
- **并发连接**：至少支持 500 个同时在线玩家（由于是 MVP，可放宽）。
- **转发延迟**：代理引入延迟应小于 10ms。
- **内存占用**：每个连接控制在大约 100KB 以内。

### 4.2 可靠性
- 代理应能长时间运行，无明显内存泄漏。
- 当后端服务器关闭时，代理应能正确处理并关闭客户端连接（无需重试）。

### 4.3 安全性
- 管理 API **无认证**（MVP 阶段简化，建议仅在局域网或受信环境测试）。
- 黑名单操作通过简单 HTTP 请求即可执行，需注意暴露风险。

### 4.4 可维护性
- 代码结构清晰，协议解析和转发逻辑分离。
- 提供简单的配置文件（如 JSON 或 TOML）设置监听端口和后端地址。

## 5. 系统架构

### 5.1 组件图
```
┌─────────────────────────────────────────────────────────────┐
│                      代理主进程                              │
│  ┌──────────────┐      ┌──────────────┐                     │
│  │ MC 代理模块  │      │ HTTP 管理模块│                     │
│  │ (TCP 25565)  │◄────►│ (端口 8080)  │                     │
│  └──────┬───────┘      └──────┬───────┘                     │
│         │                      │                            │
│         ▼                      ▼                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                全局状态（内存）                         │  │
│  │  - 在线玩家表：map[name] → {ip, since}                 │  │
│  │  - 黑名单：set<string> names, set<string> ips         │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 技术栈
| 层次       | 技术选择                       | 说明                         |
|------------|--------------------------------|------------------------------|
| 代理核心   | C++17 + Workflow               | 异步网络库                   |
| 协议解析   | 自研（参考 minecraft）           | 仅实现 1.21.11 必要包        |
| HTTP 服务  | Workflow 内置 HTTP 服务        | 提供 REST API                |
| 前端       | 纯 HTML + JavaScript + CSS     | 无构建步骤，单文件或简单内嵌 |
| 数据存储   | 内存（STL 容器 + 互斥锁）       | 无持久化                     |

## 6. 开发阶段与里程碑

### 6.1 极简 MVP 开发计划
**周期**：1 周（假设每天 4 小时投入）  
**交付内容**：
- 可运行的代理程序，配置文件支持设置监听端口、后端地址。
- 支持 1.21.11 客户端连接，正确提取玩家名。
- 内存黑名单，通过 HTTP API 动态管理。
- 静态 HTML 页面（可通过浏览器访问），展示在线玩家和黑名单，支持封禁/解封操作。
- 基本日志输出（连接、断开、封禁事件）。

### 6.2 阶段任务分解
| 天数 | 任务 |
|------|------|
| 第 1 天 | 搭建 Workflow TCP 代理框架，实现简单的 echo 测试，熟悉 Workflow 接口。 |
| 第 2 天 | 实现 Minecraft Handshake 包解析（VarInt 读取），提取协议版本和 Next State。 |
| 第 3 天 | 实现 Login Start 包解析，获取玩家名；结合 IP 实现黑名单检查；建立到后端的连接并开始转发。 |
| 第 4 天 | 实现在线玩家记录和移除；实现 HTTP 服务，提供 `/api/players` 和 `/api/bans`。 |
| 第 5 天 | 实现封禁添加和解除 API，并在封禁时踢出在线玩家。 |
| 第 6 天 | 编写前端 HTML 页面，通过 AJAX 调用 API，实现自动刷新。 |
| 第 7 天 | 测试、修复 bug、编写文档和配置文件示例。 |

## 7. 附录

### 7.1 配置示例（config.json）
```json
{
  "proxy_port": 25565,
  "backend_host": "127.0.0.1",
  "backend_port": 25566,
  "http_port": 8080
}
```

### 7.2 API 详细定义
- **GET /api/players**  
  响应：
  ```json
  [
    {"name": "Steve", "ip": "192.168.1.10", "since": 1699267200},
    ...
  ]
  ```
- **GET /api/bans**  
  响应：
  ```json
  {"names": ["Alex"], "ips": ["1.2.3.4"]}
  ```
- **POST /api/ban**  
  请求体：
  ```json
  {"type": "name", "value": "Steve"}
  ```
  响应：`200 OK` 或 `400`
- **DELETE /api/ban**  
  查询参数：`?type=name&value=Steve`  
  响应：`200 OK` 或 `404`
